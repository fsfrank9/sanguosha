排序依赖观察 (v12 同款推演):

1. **清账先行** — 遗计分牌/火攻重选/天香 ask 都是复核记录在案的偏差或
   简化, 面小且全在既有域内; 尤其遗计"逐席分配"的 UI/AI 面正是 5 人场
   分牌交互的原型, 拖到大场之后会在新变量里踩旧账。
2. **5 人场是引擎量变 + UI 质变** — 结算层 H1-H5 已按座次环写就 (预设/
   阵营/死亡奖惩/AOE 队列全部座席泛型), 剩验证与内奸胜负分支; 真正的
   增量在 UI: #allyZone 是"第三席"单席特例, 5 人要泛化为座席环渲染。
   首发做明置 5 人, 用可见身份保住结算正确性的验证抓手。
3. **可选身份在 5 人之后、暗身份之前** — "玩家恒主公"绕过了求助方向/
   玩家阵亡续跑/胜负文案一整层视角问题, 在座席面铺开后一次性翻转;
   暗身份的徽章/翻明渲染踩同一层身份 UI, L 先泛化, M 只改一次。
4. **暗身份推断殿后且自带验收抓手** — I2 已把"AI 只消费公开信息"变成
   有守护测试的工程范式, M 是同款方法论对"身份"字段的第二轮: 先立
   "零直读暗置身份"守护, 再上 aggressionLog + 行为特征的推断模型,
   准确率对随机基线可量化。3 人开放场里推断退化为常量, 故必须在 K 后。
5. **内容批独立可裁剪** — 蛊惑/军争缺口不阻塞主线, 殿后按需评估,
   宁缺毋滥红线不变。

结论排序: **J 清账 → K 5 人身份场与内奸 → L 可选身份 → M 暗身份与
推断 AI → N 内容评估**。

## 总体策略

沿用 v11/v12 全程验证过的工作规范: 每批一个独立可合的 PR、`npm run
verify` 全绿门禁、行为级回归测试、后台 merge 后再开下一批; 架构类批次
配守护测试 (裸操作/裸判断零容忍), 交互类批次引擎+UI+测试三件套同批
落地; 每阶段末盘点批对照验收标准。另加 v12 期间成文的 Git/PR 约定
(CLAUDE.md + PreToolUse hook): 推送前核对上游 PR 状态, 已合并 PR 不
复用, 每批独立 PR。

全程双红线: **1v1 与 3 人身份场行为零回归** (既有测试零改动通过) —
v12 的"1v1 红线"升级为二维, H/I 建成的多人结算与 AI 基准测试全部照跑。

## 阶段拆分

### J. v12 清账与全局合规盘点 (P0)

- **J0 全局规则合规盘点 (先行)**: 玩家实测反馈 (PR #165 合并评论) 的
  4 例定点缺陷先修:
  1. **出牌确认缺失**: 部分牌选中即直接打出、无确认步骤 — 按"有目标
     或有代价必过确认"统一 UI 交互口径, 逐牌类盘点直出白名单。
  2. **延时锦囊无懈时机反置**: 现于使用/放置时开无懈窗口; 官方时点为
     目标**判定阶段生效前**。乐/兵粮/闪电 (含国色转化乐) 统一把无懈
     窗口迁移至判定生效前, 放置时不再询问。
  3. **防具响应顺序反置**: 被杀/决斗/南蛮/万箭指定需响应时, 现先询问
     手牌响应、防具效果后置结算 — 官方语义应反过来: 免疫类防具
     (藤甲对普通杀/南蛮/万箭, 仁王盾对黑杀) 直接短路整个响应询问;
     判定类防具 (八卦) 先给发动机会, 判定成功视为已响应免出手牌,
     失败才回到手牌响应窗口。
  4. **桃合法性破口**: 满血仍可出桃、且可指定敌方座席回血 — 出牌阶段
     桃限"自己已受伤"且目标恒为自己 (濒死救援走既有求桃队列, 不受
     影响); isLegalCardTarget/canPlayCard/UI 高亮三层同步收口。
  定点修复之上跑**第三轮全量合规审计** (v10/v11 审计轮范式): 技能/
  卡牌/武器/防具逐条对照 official spec 缓存与官方规则描述, 专项覆盖
  **结算顺序与响应时机** (防具先后/无懈窗口时点/铁骑-八卦交互/AOE
  逐席顺序/判定替换时机), 产出缺陷清单入修复批; 修不完的如实记录
  降级, 不许"看起来对"。
- **J1 遗计 3p 逐席分牌**: skills 域 yiji 分配目标恒 `opponent(actor)`
  的功能偏差 → 逐点分配可选任意存活座席 (官方语义, 含自己); AI 启发
  (盟友补血线/自留) + 玩家分配面板扩席; 1v1 行为恒等 (单对手退化)。
- **J2 火攻成本挂起重选面板**: 三重巧合路径 (无懈拉锯 × 展示缓存消耗 ×
  重展示花色变化) 下, 人类玩家显式成本失效当前被自动代选同花色 —
  补 pendingChoice 挂起重选三件套 (引擎 resolver + 面板 + 行为测试);
  无同花色仍走无伤结算兜底。
- **J3 天香 ask 升级**: 伤害流暂停框架 — 伤害转移时机可挂起玩家询问
  (G2 记录: 现为铁骑式 auto/always/decline 三态), 对齐既有 ask 范式。
- **J4 工程卫生**: 风包 gid 官方页面爬取核对 (G0 唯一遗留, 临时编号
  216-223 → 官方 gid, 转写 sha256 复核); 基准慢测试分层预案 (verify
  现 ~30s 不动; K 加 4/5 人加压基准若显著拖慢门禁, 再落地快/全两档)。

### L. 可选身份 (P1)

- **L1 身份选择**: setup 身份指定/随机分配 (预设阵型内轮转), 玩家
  非主公开局路径打通; 黄天给牌按钮回归 (H8 死钮移除时留的锚点 —
  玩家为张角以外群势力席时给牌方可达); 激将/护驾求助双向化 (玩家为
  同势力代打方时被询问, H7 面板已建, 打通可达性)。
- **L2 玩家阵亡续跑**: 玩家死亡而胜负未分 → AI 续跑至终局的旁观模式
  (保持信息边界, 不明置他人手牌), 胜负文案按玩家真实身份结算 (忠臣
  以主公存活为胜等)。
- **L3 盘点批**: 任意身份 × 3/4/5 人矩阵行为测试; 1v1/3p 双红线复跑。

### M. 暗身份与推断 AI (P1-P2)

- **M1 暗置层**: 除主公外身份暗置 (状态字段 + UI 问号徽章), 死亡翻明,
  奖惩按真实身份结算 (既有逻辑); 盘点忠臣救主/立场判断等 AI 路径的
  身份直读点。
- **M2 去全知第二轮**: I2 方法论复用 — 身份读取统一路由 (明置直读 /
  暗置走推断), **"目标评估零直读暗置身份"守护测试常驻** (全局 grep
  锚点 + 行为断言)。
- **M3 推断模型**: `aggressionLog` (I3 预留数据面) + 行为特征 (对主公
  出伤害牌/救援立场/无懈方向/求助响应) → 座席×身份后验打分;
  `aiPickHostileTarget` 以期望敌意替代 sideOf 直读; 内奸策略初版
  (血线均衡骑墙, 接 K4 记录的简化)。
- **M4 验收基准**: 固定种子对局身份推断准确率对随机基线显著提升
  (量化门禁); 暗身份 AI 对"明置直读"基线的胜率差可测; 玩家侧推断
  提示 UI (可选, 按需)。

### N. 内容评估批 (P2, 按需可裁剪)

- **蛊惑 (于吉)**: 声明 → 座次质疑 → 翻验 → 生效/惩罚的多人质疑框架
  (3+ 人已具备); 先评估交互框架成本再决定接入, 宁缺毋滥。
- **军争缺口盘点**: 军争核心已随 39 张牌组数据驱动入库 (火/雷杀、
  藤甲/白银/古锭/朱雀等), 对照官方牌表清点剩余缺口与花色点数分布,
  仅接入有可靠官方 spec 来源者。
- **新扩展包评估** (林/山等): 同 spec 红线, 默认不做, 有可靠来源再立项。

## 验收标准

- J 后: 玩家实测 4 例定点缺陷修复 (含行为回归测试); 第三轮全量合规
  审计完成, 缺陷清单清零或如实降级记录; v12 记录在案三项偏差/简化
  清零 (遗计逐席分牌、火攻挂起重选、天香 ask), 风包 gid 核对落库;
  全量测试零回归。
- K 后: 4/5 人身份场可完整对局至胜负, 内奸单独获胜/内奸败北/主公方
  收紧胜利三条路径均有行为测试; 4/5 人固定种子加压基准入门禁 (零挂起/
  守恒/终局可达); 1v1 与 3 人全量测试零改动。
- L 后: 玩家以任意身份完整对局 (含阵亡后 AI 续跑至终局); 黄天给牌
  按钮真实可达; 双红线保持。
- M 后: 暗身份模式 AI 零直读暗置身份 (守护测试常驻); 推断准确率对
  随机基线显著提升且入量化门禁; 开放身份模式行为恒等 (明置直读路径
  零变化)。
- 全程: 零 npm 依赖、每 PR verify 全绿、每批独立 PR、推送前核对上游
  PR 状态、每阶段末盘点批对照本节。

## 非目标

- 不做联网对战 / 房间系统 (本地同屏 + AI 座席不变)。
- 不引入打包器 / 框架 / jsdom (零依赖红线不变)。
- 不做 6 人以上大场 (5 人封顶; 6-8 人预设留 v14 候选)。
- 不重写 UI 视觉体系 (座席环布局是排版扩展, 非视觉重绘)。
- 不承诺扩展包全量: 无可靠官方 spec 来源的技能/牌零接入。
- M 不追求身份场"完美博弈": 验收的是不作弊 + 可量化优于基线, 内奸
  高阶博弈 (装忠节奏/拆家时机) 不设胜率门槛。

## 排期与节奏

按 v11/v12 实测节奏 (每批一个 PR、后台 merge 驱动), 合计约 12-17 批:
J 4-5 (J0 盘点+修复占 2-3) → K 3-4 → L 2-3 → M 3-4 → N 0-2。顺序
执行, 阶段内小批推进;
K3 座席环渲染若发现 1v1/3p 渲染路径回归面失控, 允许拆细 (布局/交互/
面板各一批), 以批次数换确定性。每阶段末盘点批同时是"是否继续下一
阶段"的决策点, 用户可在任意盘点批后调整方向或叫停。
