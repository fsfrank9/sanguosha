# v12 方向: 结构减重 + 扩展包技能 + 多人模式 + AI 进阶

**日期**: 2026-07-05
**起点**: v11 收官 (批次 11-36 / PR #125-#150 全部已合并)
**当前状态**: 1207 断言 ✓, `npm run verify` 通过; 技能 40/审计 40, 英雄 71;
game-engine 4137 行 / dom-adapter 1559 行; 域模块 (ai/damage-dying/response/
tricks/equipment) 与面板模块 (response/prompt/mode-panels) 就位

## 缘起

v11 收尾盘点留下四个候选方向, 用户确认**四个都做**。本文档回答"以什么顺序、
各切多大、彼此怎么衔接"。关键依赖观察:

1. **加东西之前先减重** — v11 的实测教训: B 阶段把 game-engine 压到 3777 行,
   C-E 阶段加功能又涨回 4137。v12 的 G (新技能) 和 H (多人) 都是纯增量,
   若不先把"增长落点"隔离出主文件, 行数目标会再次落空。主文件剩余的
   最大块恰好是增长落点本身: 技能 trigger 函数群、判定区结算、playCard
   类型分发链。

2. **多人模式是全局性改造, 必须踩在最干净的结构上** — 座次抽象要触碰
   引擎里数百处 `'player'/'enemy'` 字面量与 `opponent()` 假设。域模块越薄、
   出口越统一 (v11 的 moveCard/手牌失去事件/装备 handler 表), 迁移面越小。

3. **扩展包技能先于多人** — 风包等技能大多 1v1 可达, 不依赖 H; 反过来,
   H 完成后 v11 留下的 reserved hook (流离/同疾) 与 1v1 退化技
   (激将/护驾/离间) 自动进入可激活状态 — 把"多人激活旧技能"作为 H 的
   收尾批, 顺序上技能池在前、多人在后正好两头受益。

4. **AI 深化殿后** — 多步 lookahead 与概率建模的收益依赖最终博弈形态;
   多人化会重写评估空间 (多对手/身份推断), 提前深做必然返工。AI 阶段
   同时是风险缓冲: 独立可裁剪, 不阻塞其他方向。

结论排序: **F 结构减重 → G 扩展包技能 → H 多人模式 → I AI 进阶**。

## 总体策略

沿用 v11 全程验证过的工作规范: 每批一个独立可合的 PR、`npm run verify`
全绿门禁、行为级回归测试、后台 merge 后再开下一批; 架构类批次配守护测试
(裸操作/裸判断零容忍), 交互类批次引擎+UI+测试三件套同批落地。
每阶段末安排一个"盘点+回归"小批, 对照验收标准后再进入下一阶段。

## 阶段拆分

### F. 结构减重第二轮 (P0, 3-4 个 PR)

- **F1 技能域拆分**: game-engine 里的 `triggerXxx` 技能函数群 (约 40 个
  技能的 handler, 主文件最大剩余块) 整体迁往 `src/engine/skills.js`,
  注册表 (`SkillRuntime.registerSkill` 调用群) 一并迁移; 引擎只留装配。
  **立规矩**: 此后新技能 handler 只进 skills.js, 配架构守护测试
  (game-engine 零 `registerSkill` 调用), 从机制上杜绝 C 阶段式回涨。
- **F2 判定区结算域**: processJudgeArea / applyJudgeAreaOutcome /
  判定挂起恢复流 (H2 快照) 迁往 `src/engine/judge-area.js`。
- **F3 playCard 分发表化**: playCard 内按 card.type/family 的长 if 链改为
  `PLAY_HANDLERS` 注册表 (基本牌/锦囊/装备各自注册), 为 H 的目标选择
  参数扩展预留统一入口。
- **F4 dom-adapter 大厅/选将拆分**: lobby + 选将网格 + 技能条渲染迁往
  `src/ui/panels/lobby-panels.js`, 主 adapter 收敛到渲染框架与注册表。
- 迁移手法沿用 B1/B2: 函数体逐行一致、deps 包装注入、双源切片更新锚点、
  全量测试零回归护航。

### G. 扩展包技能池 (P1, 4-5 个 PR)

- **G0 spec 缓存补齐**: 风包 (及后续包) 官方技能 spec 固化进
  `tests/fixtures/` (对齐 standard 包的 localSkillId + 七字段 spec 格式),
  四方一致性审计扩展到新包条目。无可靠官方来源的技能不接入 (宁缺毋滥)。
- **G1-G3 按 C 阶段范式分批接入** (每批 3-4 个, 被动/锁定优先, 交互技
  三件套殿后)。风包 1v1 可达性预判:
  - 直接可达: 神速(夏侯渊) / 据守(曹仁) / 烈弓(黄忠) / 狂骨(魏延) /
    红颜(小乔) / 雷击+鬼道(张角) / 不屈(周泰, 需濒死流程扩展) /
    天香(小乔, 1v1 转移目标恒为来源)
  - 退化或高风险, 评估后定: 蛊惑(于吉, 声明+质疑是多人机制) /
    黄天(主公技, 随 H 激活)
- 预期技能 40 → 约 50; 每批照跑四方一致性审计与守恒断言。

### H. 多人模式 (P1-P2, 7-10 个 PR, v12 主战役)

- **H1 座次抽象 (最大单项风险)**: `game.seats = ['player','enemy',...]` +
  StateRuntime 提供 seatList/nextSeat/aliveSeats; 引擎逐域用"座次环遍历"
  替换 `opponent()` 二元假设 (1v1 时 seatList 恒等旧值, 全量测试零回归
  护航); 'player' 语义收敛为"本地 UI 座席"。分 2-3 个 PR 按域推进。
- **H2 距离环与结算顺序**: distanceBetween 泛化为 min(顺时针, 逆时针) +
  马匹修正; AOE/桃园/五谷/濒死救援按逆时针座次迭代 (v11 的 responders
  队列与 order 概念直接泛化)。
- **H3 目标选择框架**: 1v1 恒定"对手即目标"的假设拆除 — 引擎侧
  playCard 增加 targets 参数与合法性校验 (攻击范围/距离/技能限制);
  UI 侧新增"选牌 → 高亮合法目标 → 点选确认"交互 (三件套同批)。
- **H4 响应链多人化**: 无懈按座次环逐位询问、决斗/借刀跨座结算、
  铁索传导多人环。
- **H5 身份场**: 首发 3 人身份 (主/忠/反) 胜负判定, 预留 5 人 (加内奸)
  的身份配置表; 身份相关技能 (主公技) 激活。
- **H6 UI 多座布局**: 座位环渲染 (2-5 人), 沿用 cream 卷轴风不重绘视觉;
  fake-DOM 行为测试同步扩展。
- **H7 多人激活收尾批**: 激将/护驾/离间接入 + 流离/同疾 reserved hook
  多人行为测试 + AI 基础目标选择 (最近威胁启发)。
- 全程红线: **1v1 模式行为零回归** (双模式并存, 1v1 测试一条不改)。

### I. AI 进阶第二轮 (P2, 2-3 个 PR)

- **I1 两步 lookahead**: 现有单步模拟 (clone + play + eval) 扩展为
  "我方行动 → 对手最优回应 → 评估", 复用 aiSimulateCardPlay 管线;
  节点数用候选剪枝控制 (只展开 top-N 候选)。
- **I2 可见信息计数建模**: 弃牌堆/已亮判定/已知手牌的剩余牌型计数,
  替换 aiEstimateShaCount 的均匀假设 (闪概率/桃存量估计)。
- **I3 多人目标评估**: 身份推断简版 (按攻击行为累计敌意分) + 集火/
  保护决策; 依赖 H, 若 H 延期可先落 I1/I2 (1v1 同样受益)。

## 验收标准

- F 后: game-engine ≤2200 行 (技能群/判定域迁出后的现实目标),
  dom-adapter ≤1200 行; game-engine 零 registerSkill 调用 (守护测试);
  全量测试零回归。
- G 后: 技能 ≥50, 新包条目全部通过四方一致性审计; spec 无来源的技能
  零接入。
- H 后: 3 人身份场可完整对局 (开局→身份胜负判定); 1v1 全量测试零改动
  通过; 目标选择三件套 (引擎/UI/测试) 齐备。
- I 后: AI 在 1v1 自对弈基准中对 v11 版 AI 胜率 ≥55% (用固定种子
  批量对局测试衡量)。
- 全程: 零 npm 依赖、每 PR verify 全绿、每阶段末盘点批对照本节。

## 非目标

- 不做联网对战 / 房间系统 (本地同屏 + AI 座席)。
- 不引入打包器 / 框架 / jsdom (零依赖红线不变)。
- 不重写 UI 视觉体系 (多座布局是排版扩展, 非视觉重绘)。
- 不做 5 人以上大场与军争篇卡牌 (v13 候选)。
- 不承诺全部扩展包: 只接入有可靠官方 spec 来源且非多人专属的技能。

## 排期与节奏

按 v11 实测节奏 (每批一个 PR、后台 merge 驱动), 合计约 16-22 批:
F 3-4 → G 4-5 → H 7-10 → I 2-3。顺序执行, 阶段内小批推进;
H1 座次抽象若在中途发现回归面失控, 允许拆得更细 (每域一批),
以批次数换确定性。每阶段末的盘点批同时是"是否继续下一阶段"的
决策点, 用户可在任意盘点批后调整方向或叫停。
